why local

Vercel shows it.
Your machine runs it.

The hosted deployment at /foreman is a view-only tour — serverless has no long-lived worker, no git, and no persistent disk. The actual multi-agent engine needs all three. Locally, you get the real system.

real seat

Claude decides

The implementer & reviewer run on claude-opus-4-8 — not a scripted stub.

blast radius

Git worktree

Each task gets its own branch + working tree. A bad run is git worktree remove.

on the record

SQLite ledger

Tasks · runs · events persist on disk and survive across sessions.

truth signal

Your real tests

Green means your suite passed — the one thing the model can't talk past.

before you start

Four things to have ready.

runtime

Python 3.10+

A recent CPython. A virtualenv keeps it clean.

target

A git repo

The project Foreman will develop in. Worktree isolation needs it to be a real repo.

credentials

An API key

ANTHROPIC_API_KEY for Claude in the decision seat.

the gate

A test command

The objective truth signal, e.g. pytest -q. This is what "done" means.

step 01 — install

Clone, venv, install.

$ git clone <your-fork> resume_agent && cd resume_agent
$ python3 -m venv .venv && source .venv/bin/activate
$ pip install -e '.[web,llm]'
  • web → the browser control panel (starlette + uvicorn).
  • llm → the anthropic SDK that puts Claude in the seat. (Foreman itself only needs anthropic from this extra.)
step 02 — credentials

Give it a key.

$ export ANTHROPIC_API_KEY="sk-ant-..."
# …or drop it in .env (gitignored) — same effect

Foreman's LLMPolicy constructs anthropic.Anthropic(), which reads the key straight from the environment. No key yet? You can still drive the whole loop offline with --policy mock — free, no network (next slide).

step 05 — a real run

Point Claude at real code.

in the browser

New run → a project folder

Pick "A project folder on this computer", paste the path, choose Claude + a model, and run.

from a shell

One command

The same run, scriptable. The flags below are the whole contract.

$ python -m foreman run \
--title "Fix flaky retry in client.py" \
--repo /path/to/your/git/repo \
--policy llm --model claude-opus-4-8 \
--test-cmd "pytest -q"
Two non-negotiables: the --repo must be a git repo (that's the worktree's blast radius), and --test-cmd must be your real truth signal. Omit it and Foreman falls back to python check.py — fine for the demo, wrong for your project.
under the hood

Plan. Isolate.
Iterate. Review.

  • Orchestrator (plain Python) turns the task into work and opens a git worktree on foreman/task-N.
  • Implementer (Claude) reads files, edits code, and runs your tests — looping red → fix → green.
  • Reviewer (fresh-context Claude) reads the diff and returns approve or request_changes — which routes back to the implementer.
  • Green + approved → pr_ready, committed to the branch. Watch every step live in the UI (it auto-refreshes).
The implementer's tools are read_file · edit_file · run_tests; the reviewer's are read_file · submit_review. There is no merge or push tool — the irreversible step simply isn't in reach.

The irreversible step

Open the Review queue
Read the diff, tests, and verdict
Click Merge (I approve) — records it
Then actually land the branch ↓
$ git checkout main && git merge foreman/task-3
# the branch is already waiting in your repo
The Merge button records that you approved — Foreman never runs git merge or git push. That capability does not exist in its tool set, by design.
audit trail

Every run is on the record.

One SQLite file is the whole ledger — tasks · runs · events (append-only). Every tool call, test result, and verdict is auditable after the fact. The Stats page turns it into the numbers that matter.

$ python -m foreman tasks # status of every task
$ python -m foreman stats # success % · avg turns · cost $ · human-touch
$ sqlite3 data/foreman/foreman.sqlite3 '.tables'

Or just open /stats and /runs in the control panel.

if a run parks itself

When it stalls.

needs_human · "anthropic SDK…"
Install the extra and set the key: pip install -e '.[web,llm]' + export ANTHROPIC_API_KEY=…
tests never go green
Your --test-cmd must actually pass in that repo (deps installed). Open the task page to read the diff and the failure.
no diff · "not a git repo"
--repo must point at a real git repo — that's what the worktree branches from.
stops mid-run
Bounded by max_turns + a token budget — it parks as needs_human rather than spinning forever.
