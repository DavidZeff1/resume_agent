In depth — What jobagent is

jobagent is a local, single-user agent that prepares the whole job-application pipeline and stops at the one irreversible step. It runs on your machine, stores everything in a plain SQLite file, and is driven by a jobagent CLI with an optional local web UI.

The core pipeline needs only ordinary libraries — httpx, PyYAML, Jinja2, python-docx, beautifulsoup4. Claude-backed scoring and the Agent-SDK loop are optional extras; nothing essential depends on an API key.

In depth — The recoverable / unrecoverable split

Sourcing, scoring, tailoring and prep can all be wrong cheaply: a skipped or mis-ranked job costs nothing and can be revisited. So they run fully autonomously, optimised for throughput.

Submission is a one-way door — a botched application to a company you wanted can't be retracted, in a small, reference-driven market. It is never optimised for volume.

The metric isn't applications sent; it's human-touch rate — the share of prepared apps that still need you. It starts high by design and is lowered later, only per ATS that proves clean.

In depth — One deterministic pass

jobagent run executes source → score → tailor → prep → followup in a single deterministic pass (pipeline.run_once) and parks ready applications in the review queue.

Source pulls from your finite watchlist; score shortlists or skips with a rationale; tailor attaches the best stored resume and cover letter; prep pre-fills the form and lists what's left for you. Track, follow-up and schedule run on the back half, after you submit.

The same five stage functions back both this loop and the optional Agent-SDK loop, so behaviour is identical whichever drives.

In depth — Prep, the guardrail made concrete

build_prefilled() copies a fixed set of safe facts straight from your stored profile and the selected materials — name, contact, links, the résumé and cover-letter files. Nothing else.

Everything judgement-based becomes an unanswered_fields entry routed to you. The categories in NEVER_AUTOFILL_KINDS — motivation, salary, work-auth, demographic, free-text — never enter prefilled data, even when your profile holds a value (it's offered only as a suggestion).

assert_guardrails() raises if a forbidden key ever crosses that line, and a unit test fails the build if it regresses.

In depth — Why it structurally cannot submit

The guardrail isn't a policy — it's a missing capability. pipeline_agent.py hands Claude exactly six tools (source, score, tailor, prep, followup, status) and registers no submit tool; Bash, Write and Edit are explicitly disallowed too.

The CLI review submit and the web "record submission" button only flip a status to submitted and stamp the time — neither posts anything to a careers site. A model told to "just apply" still can't: there is no code path from here to a form.

In depth — The status state machine

Every job moves through a fixed graph defined once in models.py. validate_transition(current, target) rejects anything outside the allowed set with InvalidTransition — so a jump like discovered → submitted is impossible.

skipped is terminal-but-revisitable (a re-score can pull it back). submitted is reachable only by a human action. Afterwards, outcomes flow between each other as news arrives; rejected is terminal and an offer can still fall through.

The CLI, web UI and agent loop all route through the same validator.

In depth — Each guardrail, in code

1· No submit tool exists anywhere. 2· build_prefilled() copies only stored profile fields; assert_guardrails() raises if a forbidden key leaks in. 3· NEVER_AUTOFILL_KINDS — motivation, salary, work-auth, demographic, free-text — always route to you; a stored value is offered as a suggestion, never auto-filled.

4· PoliteClient enforces robots.txt, per-host spacing, caching and backoff. 5· the API key is a lazy os.environ read — never stored, never logged. 6· an append-only events table records every action.

A dedicated test file fails the build if any of these regress.

In depth — Package layout

A clean Python package, one module per pipeline stage, each independently testable. Core files own config, the SQLite connection, the state machine, every DB query and the audit log.

Stage packages — source/ score/ tailor/ prep/ review/ track/ — hold the real logic; state/ holds the human-authored inputs. The CLI, the Starlette web app and the Agent-SDK loop are all thin wrappers over the same functions, so the guardrails hold whatever the entry point.

api/index.py adapts the same app for Vercel, with a diagnostic fallback that serves the traceback instead of an opaque 500.

In depth — The SQLite schema

Seven tables in one file. profile is a single row — the only facts the agent may use. companies is your finite watchlist; each owns many jobs, deduplicated by UNIQUE(url).

A shortlisted job gets exactly one applications row (UNIQUE(job_id)), which references the chosen resume_variants and cover_letters and stores prefilled_data + unanswered_fields as JSON.

events is an append-only audit log — polymorphic over entity_type + entity_id, so every state change is recoverable after the fact.

In depth — Two scorers, one interface

Scoring is a pluggable Scorer. The heuristic backend is pure-Python keyword matching across eight track dictionaries, title hits weighted double, with mild seniority and skill nudges — deterministic, offline, and what every test uses.

The Claude backend puts the profile + instructions in an ephemeral-cached system block (stable across a run) and varies only the job text per call, so repeated scoring hits the prompt cache. It returns a score, a suggested track and a short rationale, and is instructed never to invent candidate facts.

backend: auto uses Claude if a key is present, else the heuristic — so nothing requires credentials.

In depth — Sourcing, polite by construction

Sourcing prefers official, sanctioned JSON: the Greenhouse and Lever job-board APIs and Workable's public widget endpoint. Only true HTML boards (generic/Comeet) get parsed — and there it prefers schema.org JobPosting JSON-LD, falling back to a light anchor heuristic.

Every fetch passes through one choke point, PoliteClient: robots.txt for non-sanctioned URLs, per-host rate limiting, on-disk caching with a TTL, and retries with backoff that honour Retry-After.

No LinkedIn, no Glassdoor, no open-web crawl — the watchlist is finite and human-defined.

In depth — A view over the same state

jobagent web serves a local Starlette app that is a thin view over the very same SQLite state the CLI uses: each request opens its own context, runs the same repo + state-machine code, and closes it. Every guardrail holds identically.

There is intentionally no route that submits to an ATS; /review/{id}/submit only records that you did. Every mutating action is crash-proofed — bad input or an illegal transition becomes a clear in-page message, never a 500.

Optional HTTP Basic auth (JOBAGENT_WEB_PASSWORD) gates it, since it shows personal data.

In depth — Running it locally or on Vercel

Easiest path: jobagent web on localhost, reached remotely via a Tailscale or cloudflared tunnel — a plain SQLite file and zero infrastructure.

For serverless, api/index.py exposes the app as a top-level ASGI app Vercel can detect. It needs a hosted libSQL/Turso database (JOBAGENT_DB_URL) so state survives between invocations, a writable /tmp data dir, and a password gate.

jobagent demo seeds a realistic offline example and runs the pipeline, so the review queue is populated immediately — no key, no network.

In depth — Earned autonomy (build plan §9)

The success metric is human-touch rate — the share of prepared applications that still need you before submit. v1 starts at 100%: you review everything.

It comes down only by whitelisting specific ATS types that prove clean — stable forms, no CAPTCHA, no surprise free-text — and even then never for an application that still has any unanswered_fields.

So the curve has a floor. The human is never removed from the irreversible step by fiat; autonomy is earned, per site, with evidence.

In depth — The line it builds to

The architecture gives you maximum leverage on the recoverable ~90% of the work while keeping you on the irreversible ~10%.

Success is measured by human-touch rate trending down — but only by whitelisting specific ATSs that prove clean over time, never by removing the human from submit up front. Per-ATS auto-submit, smarter follow-ups and analytics are deliberately out of scope for v1, and each must preserve every guardrail.
