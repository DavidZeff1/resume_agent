In depth — the recursive idea

jobagent is built on one rule: automate everything recoverable, keep a human on the one irreversible step (submitting an application). Foreman applies that same rule to the act of building jobagent.

Writing code on a branch is recoverable — a bad commit is git worktree remove and it never happened. Merging to main enters shared history and may ship. So the agents do everything up to a pull request; the human merges.

The gate that keeps the recoverable loop honest is the thing jobagent already has: its 62 passing tests. Agents iterate until the suite is green — not until they feel done.

In depth — who does what

The orchestrator is plain Python, not a model — you don't need an LLM to manage a queue, a git worktree, and a state machine. It mirrors jobagent's deterministic run_once.

The implementer and reviewer are the only model loops. The implementer edits in an isolated worktree and iterates against run_tests until green; a fresh-context reviewer then approves or sends it back with findings.

Only a green, reviewed diff reaches the human as a pull request.

In depth — the agent taxonomy

Each worker is an autonomous agent (owns its loop), reactive (responds to test results), proactive (keeps acting toward the goal), and social (coordinates through the blackboard) — the classic agent properties.

They are goal-based / deliberative: the goal is an explicit terminal state (tests green + approved), not a fixed reflex. Coordination is a blackboard architecture — agents never call each other; they read and write shared state and react to changes in it.

The orchestrator is a control agent — deterministic Python, not an LLM — sequencing the workers through the state machine.

In depth — PEAS & the decision seam

PEAS for a worker — Performance: a green, reviewed diff in few turns · Environment: the workspace + task · Actuators: the edit / bash / run_tests tools · Sensors: file reads, the diff, the test exit code.

decide is a pluggable Policy. The deterministic MockPolicy drives the whole loop offline so it's testable; LLMPolicy runs the manual agentic loop over the Messages API with Claude in the seat — the one part that needs credentials.

In depth — why an objective gate

The single most important design decision for a code-writing agent system is the feedback loop. If an agent grades its own work, it will confidently declare success on broken code — that's exactly where it fabricates.

So "done" is defined by something the agent can't talk its way past: pytest green, plus mypy and ruff clean. run_tests is a dedicated tool the harness owns, so the result is parsed and recorded — the reviewer is never handed an unverified diff.

Red routes back to the implementer; the human only ever sees green.

05 The truth signal

"Done" means the tests pass.

In depth — capability is the guardrail

The agents get Anthropic's schema-less bash and text_editor tools (you declare them, you execute them locally) plus a dedicated run_tests. That's enough to read, edit, and verify code.

What they don't get is a merge or push tool — so the agent cannot perform the irreversible action, the same way jobagent's loop has no submit tool. Security checks (path confinement, command allowlist) live in the tool executors, in one place.

In depth — the hand-rolled loop

Each model role is a manual agentic loop over the Messages API: call with tools, branch on stop_reason, execute tool calls, feed results back, repeat until end_turn. You own the loop — that's the point of the project.

Stable context (repo map, conventions, the test command, the task's acceptance notes) goes in a prompt-cached system block, so every iteration re-reads it at ~0.1× cost. The biggest lever in a multi-turn loop.

In depth — the real UI

This is a real interface now — foreman web serves it (Starlette) as a thin view over the same SQLite ledger the CLI writes, exactly like jobagent's UI. The screen below is illustrative, not a screenshot.

The review queue is the heart of it: each task shows its prepared diff, the green test result, and the runs/cost — and the Merge button performs the one human action. No route merges or pushes to a real remote.

It's a full control panel for non-coders: start a run from a form, watch the agents work via live status (the page auto-refreshes), and merge — all in the browser. Launch with foreman web, or just double-click Foreman.command.
